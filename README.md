# Nokki Nokk рџ‘ЃпёЏрџ‘ЃпёЏ

Nokki Nokk is a secure, temporary, and anonymous file and text sharing web application. Users can upload files, text snippets, or links, set parameters, and retrieve them on other devices instantly using a 5-character secure alphanumeric PIN or a direct share link. All shared data automatically and permanently expires after **2 minutes**, erasing database entries, cache mappings, and physical files on disk.

---

## рџљЂ Key Features

*   **P2P WebRTC Direct Sharing**: Supports sending files up to **50 GB** directly browser-to-browser, bypassing server storage for absolute speed and privacy.
*   **Resumable 30s Reconnection**: If the network connection drops during a P2P transfer, the system enters a reconnecting state for up to 30 seconds. Upon reconnecting, it resumes from the last successfully transferred byte offset.
*   **Direct Disk Streaming**: Uses the File System Access API (`showSaveFilePicker`/`showDirectoryPicker`) to stream large files directly to the local disk, avoiding RAM overhead and preventing browser crashes.
*   **Backpressure Flow Control**: Throttles the WebRTC transmission rate based on the network buffer (`bufferedAmount`), preventing packet loss and buffer overflows.
*   **Multi-Format Sharing**: Supports uploading multiple files (up to **100MB** total size in standard mode), text snippets, or link sharing.
*   **Zero Accounts Required**: No signup, sign-in, or cookies. Shared resources are linked solely to a secure transient PIN.
*   **Dynamic Countdown**: Real-time remaining time displays for both the sender and the receiver.
*   **Autogenerated QR Code**: Instantly generate and scan QR codes for mobile retrieval.
*   **Auto Clean-up Worker**: A background worker runs every 60 seconds to completely wipe expired data from the database and unlink files from the server's disk storage.

## рџ”’ Security Hardening & Mitigations

*   **Single-Receiver P2P Locking**: P2P PINs are strictly single-use. Once verified by the receiver, the PIN is deleted from Redis to block anyone else from connecting. The session is locked in Redis under the recipient's unique `receiverClientId` to secure reconnection attempts.
*   **Cryptographically Secure Random PINs**: PINs are generated using Node's `crypto.randomInt()`. The PIN space uses 5-character alphanumeric values (`[A-Z2-9]`, excluding confusing characters `O`, `0`, `I`, `1`), expanding the key space to **33,554,432 combinations** to prevent collisions and predictive attacks.
*   **Stored XSS Prevention**: Links are validated against strict protocols (only `http:` and `https:` allowed). Unsafe URI schemes like `javascript:` are rejected by the backend and blocked by defense-in-depth checks on the frontend before rendering.
*   **Download Authentication**: Accessing file downloads strictly requires passing the active PIN as a validation query parameter (`?pin=XXXXX`). If the PIN is missing or incorrect, downloads are rejected with a `403 Forbidden` error.
*   **IP-Based Rate Limiting**: Protects PIN retrieval (max 5 attempts per minute) and resource uploading (max 10 uploads per minute) to block brute-force and Denial of Service (DoS) attacks.
*   **Secure Client IP Lookup**: Uses clean Express `req.ip` resolution mapped through a configurable `TRUST_PROXY` setting to prevent rate-limiter bypasses via header spoofing.

## рџ› пёЏ Tech Stack

*   **Frontend**: React (Vite), WebRTC (PeerConnection & DataChannel), File System Access API, Lucide Icons, QRCode rendering, custom Glassmorphism CSS.
*   **Backend**: Node.js, Express, Socket.io (WebRTC signaling), Multer (multipart form parsing).
*   **Database**: PostgreSQL (relational metadata, schema uses `ON DELETE CASCADE` constraints).
*   **Caching/Limiting/Sessions**: Redis (in-memory rate limiters, active PIN-to-upload mappings, and P2P connection session cache).

## вљ™пёЏ Configuration

Copy the sample environment variable templates in the project to set up configurations:

### Root Level (Docker Compose)
Copy `.env.example` to `.env` in the root folder:
```bash
# Database Configuration
DB_HOST=host.docker.internal
DB_PORT=5432
DB_USER=postgres
DB_PASSWORD=your_password
DB_NAME=nokkinokk

# Cache Configuration
REDIS_HOST=host.docker.internal
REDIS_PORT=6379
REDIS_PASSWORD=

# Server Settings
PORT=5000
NODE_ENV=production
UPLOAD_DIR=./uploads
```

### Backend Configuration
Copy `backend/.env.example` to `backend/.env`:
*   Specify connection variables to your local/VPS Postgres and Redis instances.
*   Configure `TRUST_PROXY` (e.g. `loopback` or specify your Nginx gateway subnet) to securely track client IP addresses.

### Frontend Configuration
Copy `frontend/.env.example` to `frontend/.env`:
*   Specify `VITE_API_URL` during development if running servers on custom ports.

## рџ’» Running the Application

### рџђі Option A: Using Docker Compose (Recommended for Production)
Ensure Docker and Docker Compose are installed, then run at the root directory:
```bash
docker-compose up --build -d
```
*   The **Frontend (Nginx reverse proxy)** will be exposed at `http://localhost:8080` (or configured VPS port).
*   The **Backend (Express API)** will be exposed at `http://localhost:5000`.

### рџ’» Option B: Manual Local Setup (Development)

#### 1. Start Databases
Ensure PostgreSQL and Redis are running locally. Create a database named `nokkinokk`.

#### 2. Start Backend
```bash
cd backend
npm install
npm run dev
```

#### 3. Start Frontend
```bash
cd frontend
npm install
npm run dev
```
Open your browser at `http://localhost:3000`.
